# Release checklist — Advanced Permissions for Umbraco

Repo-specific release notes that complement the generic **`nuget-pre-release`** skill. The skill
owns the phase-by-phase workflow; this file records the things only this package knows. Read this
first when preparing a release.

## Release model

- **Personal GitHub package** (owned by Luuk Peters), **not** a Proud Nerds package.
- Published from **GitHub** (GitHub Actions), versioned by **MinVer** from a `v`-prefixed git tag
  (`MinVerTagPrefix=v`, auto-increment minor, minimum `1.0`). Cutting the release = creating and
  pushing the tag.
- **Phase 5 (cut the release):** done **manually in GitHub** by default, for control. Ask before
  the assistant does anything that tags/pushes.
- **Phase 6 (post-release):** **N/A.** Do **not** invoke `nuget-post-release` — that skill is for
  Proud Nerds packages on Azure DevOps (Jira / Confluence / manual version bump). This package has
  none of that; the next version comes from the next tag.

## Release notes

`publish.yml` creates the GitHub release with `softprops/action-gh-release@v2` and
`generate_release_notes: true`, so the release body is **auto-generated by GitHub from merged PRs
since the previous tag**. There is no `CHANGELOG.md` and no `<PackageReleaseNotes>` — by design.

Gotcha: because auto-notes diff against the *previous tag*, a release whose previous tag was a
prerelease (e.g. `v18.0.0` following `v18.0.0-beta.2`) gets thin auto-notes covering only the
commits since that beta. For a meaningful first-stable release, **curate the release body by hand**
(cover the whole line) and let the auto-generated PR list sit beneath it.

## Dependency hygiene (this package)

- Central versions live in
  [`Directory.Packages.props`](Directory.Packages.props). Before a stable release, the Umbraco
  package floors (`Umbraco.Cms*`, `Umbraco.Cms.Persistence.EFCore`) and `uSync` must move off
  their `-rc` floors to the **stable** floor (e.g. `[18.0.0,19.0.0)`), not stay at
  `[18.0.0-rc2,19.0.0)`. See the skill's Phase 1 for why a prerelease floor keeps resolving to
  the rc even after stable ships.
- `Microsoft.EntityFrameworkCore.*` is pinned to exactly match the version Umbraco references —
  re-check it against the **stable** Umbraco release, not the rc, to avoid MSB3277 conflicts.

## Generated API client (hey-api)

The TypeScript API client (`client.gen.ts` and the generated `api/` files) is produced by
**`@hey-api/openapi-ts`** via `npm run generate-client`, which hits a running TestSite's
`…/openapi/management.json`. hey-api is **notorious for breaking changes between versions**, so:

- It is **pinned** (currently `^0.97.0`, resolved to `0.97.3`). Don't `npm update` it casually — a
  minor bump can change the generated output and break the build. The lockfile keeps it stable; a
  plain `npm install` (as done for a dependency bump) does **not** move it.
- The **committed client is what ships** — regenerating is **not required** for a release. Only
  regenerate deliberately, and when you do, re-run `npx tsc --noEmit` + `npm run build` and review
  the diff, since both a hey-api version change *and* a host OpenAPI-spec change can alter the output.
