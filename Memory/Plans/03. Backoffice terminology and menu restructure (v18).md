# 03. Backoffice terminology and menu restructure (v18)

**Status**: Draft — awaiting user review
**Date**: 2026-06-21
**Target**: Umbraco 18
**Follow-up**: Port applicable changes to the Umbraco 17 line (separate task — see "Out of scope").

## Goal

Make the Advanced Permissions backoffice navigation consistent and self-explanatory now that the Library section has doubled the number of screens. The package currently exposes **four editors and four viewers** (eight menu items) grouped by *mode* (an "Editors" group and a "Viewers" group), with inconsistent names and four identical `icon-eye` viewers. This restructure regroups everything **by domain**, applies a uniform naming scheme, and gives each item a domain-encoded icon.

No behaviour, API, persistence, or resolution logic changes. This is a navigation, labelling, and iconography change only.

## The problem

Under the hood the feature set is a clean **2 × 2 × 2 matrix**:

| | **Node permissions** (Allow/Deny verbs on tree nodes) | **Type-level permissions** (currently create/insert filtering; future of-type verbs) |
|---|---|---|
| **Content domain** | Content Permissions Editor → Access Viewer | Document Type Permissions Editor → Insert Options Viewer |
| **Library domain** | Library Permissions Editor → Library Access Viewer | Library Element Type Permissions → Library Insert Viewer |

= 2 domains × 2 aspects × 2 modes = 8 leaf screens. The menu groups them by *mode*, forcing the user to mentally re-pair each viewer with its editor across two groups.

Concrete terminology defects today:

1. Inconsistent "Editor" suffix — three editors have it; "Library Element Type Permissions" does not.
2. Inconsistent domain prefix — "Content Permissions Editor" vs. its viewer "Access Viewer" (no domain), while Library prefixes both sides.
3. Editor/viewer vocabulary mismatch — "Document Type Permissions" pairs with "Insert Options Viewer"; "Library Element Type Permissions" pairs with "Library Insert Viewer".
4. "Insert" (content) vs. "Create" (library) for the same concept.
5. All four viewers share `icon-eye` — visually indistinguishable.
6. "Permissions" is overloaded — node verbs in some names, type-creation in others.

## Design decisions

- **D1 — Group by domain, not by mode.** Replace the "Editors"/"Viewers" groups with four domain groups. Rationale: pairs the viewer with its editor and matches how users think ("I'm working on Library permissions").
- **D2 — Editor and viewer stay as separate menu items (not workspace tabs).** Rationale: users frequently want the editor and its viewer open side by side; separate items allow that, workspace tabs would not.
- **D3 — Four symmetric groups, short item labels.** Group headers carry the domain; item labels are just the function. Mirrors the underlying matrix and is the most scannable. Accepted trade-off: the label "Permissions Editor" repeats across groups (the header + icon disambiguate).
- **D4 — Type-level aspect is NOT named "insert"/"create".** These editors today filter the create lists via a package-invented "insert"/"create" verb, but are intended to gain real of-type verbs (e.g. `DeleteOfType`, `MoveOfType`) that behave like ordinary permissions. Names must not bake in "insert"/"create". This makes labelling fully uniform: every editor is **"Permissions Editor"**, every viewer is **"Access Viewer"** — both terms survive the future verbs. (Consistent with the existing manifest note about future of-type verbs surfacing in the Access Viewer's reasoning chain.)
- **D5 — Domain-encoded icons (Scheme A).** Each item's icon is its domain glyph; the editor and viewer in a group share it, which visually binds the pair. The four viewers stop being identical eyes because each now carries its domain glyph. All four glyphs verified present in the Umbraco icon registry.
- **D6 — Full descriptive page headlines.** Menu labels are short ("Permissions Editor"), but each workspace's H1 keeps the **full** name ("Content Permissions Editor", "Document Type Access Viewer", …) so a short menu never yields a context-less page.
- **D7 — Internal aliases / entityTypes unchanged.** Identifiers like `uap-doc-type-create-audit` and `UAP.Workspace.AccessViewer` are never user-visible; renaming them is pure churn and risk. Only user-facing labels, icons, and grouping change.

## Final backoffice layout (Users section)

```
USERS                        ← native, unchanged
  Users
  User Groups

CONTENT                      ← UAP.SidebarApp.Content            (weight 60)
  Permissions Editor   icon-document
  Access Viewer        icon-document

DOCUMENT TYPES               ← UAP.SidebarApp.DocumentTypes       (weight 55)
  Permissions Editor   icon-diploma
  Access Viewer        icon-diploma

LIBRARY                      ← UAP.SidebarApp.Library             (weight 50)
  Permissions Editor   icon-globe
  Access Viewer        icon-globe

LIBRARY ELEMENT TYPES        ← UAP.SidebarApp.LibraryElementTypes (weight 45)
  Permissions Editor   icon-thumbnail-list
  Access Viewer        icon-thumbnail-list
```

Each group = one `sectionSidebarApp` (kind `menu`) + one `menu` holding its two `menuItem`s — the same mechanism the current "Editors"/"Viewers" groups use, scaled from two groups to four. The eight workspaces are untouched; only the menu they hang under, their label, and their icon change.

## Naming map (old → new)

| Old menu label | New: group › item | Workspace H1 (full, D6) | Icon (D5) |
|---|---|---|---|
| Content Permissions Editor | Content › Permissions Editor | Content Permissions Editor | `icon-document` |
| Access Viewer | Content › Access Viewer | Content Access Viewer | `icon-document` |
| Document Type Permissions Editor | Document Types › Permissions Editor | Document Type Permissions Editor | `icon-diploma` |
| Insert Options Viewer | Document Types › Access Viewer | Document Type Access Viewer | `icon-diploma` |
| Library Permissions Editor | Library › Permissions Editor | Library Permissions Editor | `icon-globe` |
| Library Access Viewer | Library › Access Viewer | Library Access Viewer | `icon-globe` |
| Library Element Type Permissions | Library Element Types › Permissions Editor | Library Element Type Permissions Editor | `icon-thumbnail-list` |
| Library Insert Viewer | Library Element Types › Access Viewer | Library Element Type Access Viewer | `icon-thumbnail-list` |

## Implementation surface (files affected)

- **`Client/src/manifests.ts`** — replace the two `sectionSidebarApp` + two `menu` entries with four of each; reassign each of the eight `menuItem`s to its domain menu; set the domain icon and the new (shared) label key on each item; set group-header label keys and weights per the layout above.
- **`Client/src/localization/en.ts` + 27 other locale files** — update label values. Because all editors share the label "Permissions Editor" and all viewers "Access Viewer", introduce two shared menu-item label keys plus four group-header keys; update the workspace headline keys to the full names in the table; reword the two redirect messages. en + nl authored properly; the other 26 updated mechanically and any needing a native speaker flagged.
- **Workspace root elements** — confirm each uses the full-name headline key (some current keys already do, e.g. `editorHeadline`; the viewer/audit/element headlines need their values changed to the full names above).
- **Redirect components** (`uap-granular-permission-redirect`, `uap-element-granular-permission-redirect`) — their messages name the old editors; reword to point at "Content › Permissions Editor" / "Library › Permissions Editor".
