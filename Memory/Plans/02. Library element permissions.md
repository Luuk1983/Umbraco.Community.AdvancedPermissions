# 02. Library (element) permissions

**Status**: Approved for implementation
**Date**: 2026-06-15
**Target**: Umbraco 18 (verified against 18.0.0-rc2 source at `C:\GitHub\UmbracoVersions\v18`)

## Goal

Extend the Advanced Security package to govern Umbraco 18's new **Library** section with the same model the package already applies to content. "Full parity" means two capabilities:

1. **Element-item access** — Allow/Deny per role per node, with tree inheritance, over the library's elements and folders (the analogue of content-node permissions).
2. **Element-type create-filtering** — which element types each role may create in the library (the analogue of the existing document-type create-filtering).

The package must remain the single authority for both **API authorization** and **backoffice UI action visibility**, exactly as it is for content.

## Scope

**In scope**
- Element-item + element-folder permission management (one editor, mirroring the content editor).
- Overriding element + element-container enforcement so denied actions are blocked at the API.
- Replacing the native element/element-folder user-group permission UI (toggles, granular picker, conditions) with our own, mirroring what the package does for documents.
- Element-type create-filtering in the library (reusing the existing Doc Type subsystem).
- A generic, reusable frontend "permission-target descriptor" so editors can express per-verb/per-node-kind applicability (drives elements now; reusable for future entity types).

**Out of scope (deferred, deliberately)**
- Full type-level verb suite for document types (only create-of-type remains; build verb-agnostic so it can be added later as a focused feature with its own type-vs-node combination spec). See decision **D7**.
- Retrofitting the existing Content / Doc Type editors onto the new generic frontend components (they keep working as-is; they can adopt the components in a later pass).
- Umbraco 19's "pick shared blocks in Block List" integration.

### The #22351 bug applies to elements (unfixed in RC2)

Issue [#22351](https://github.com/umbraco/umbraco-cms/issues/22351) (fixed for documents via PR #22400) recurs for elements:
- `DocumentPermissionMapper.AggregatePresentationModels` → `IContentPermissionService.GetPermissionsAsync` (**fixed**).
- `ElementPermissionMapper.AggregatePresentationModels` → `IUserService.GetPermissionsForPath` (**native store, bypasses `IElementPermissionService`**).
- `GetElementPermissionsCurrentUserController` → `IUserService.GetElementPermissionsAsync` (native).

`/user/current` is built by `UserPresentationFactory.GetAggregatedGranularPermissions` → `mapper.AggregatePresentationModels` (`UserPresentationFactory.cs:339`), which the native element condition reads.

**Consequence:** overriding `IElementPermissionService` gives correct API enforcement (the auth handler routes through it via `ElementPermissionAuthorizer`), but **not** UI visibility. Unlike documents, we cannot piggyback on the native current-user endpoint (it's also native for elements). Our custom element conditions must call **our own** effective-permissions API. An upstream follow-up issue to #22351 has been drafted separately.

## Architecture decisions

### D1 — Reuse the node-permission resolution core verbatim
`ResolutionEngine`, `AdvancedPermissionEntry`, `PermissionResolutionContext`, `PermissionResolver`, `EffectivePermission`, `PermissionReasoning`, `PermissionState`, `PermissionScope`, `ResolutionEntry` are verb- and dimension-agnostic with **no content-specific fields**. Element-item permissions reuse them with zero duplication. The real "new shared layer" investment is on the **frontend** (D4), not a backend re-abstraction.

### D2 — One `ElementPermission` table, canonical-verb storage
- A single new table `ElementPermission` (mirrors `AdvancedPermission`'s columns/indexes) holds entries for both elements and folders. Keep it isolated from the content table (clean cache invalidation, cleanup, queries; mirrors the existing Content/DocType split).
- **Storage uses the `Umb.Element.*` verbs as the canonical set (9).** The folder enforcement adapter maps its 5 container verbs onto that canonical set when resolving:
  `Umb.ElementContainer.{Read,Create,Update,Delete,Move}` → `Umb.Element.{Read,Create,Update,Delete,Move}`.
- Effect: one stored entry serves both families. A folder's `descendants` Read=Allow grants read to descendant sub-folders (folder adapter maps container→canonical) **and** descendant elements (element adapter resolves canonical), with no entry duplication. The 4 element-only verbs simply have no folder-adapter mapping. Create governs both child kinds at a location with one entry.

### D3 — Two thin enforcement overrides sharing one resolution
- `AdvancedElementPermissionService : IElementPermissionService` — resolves `Umb.Element.*` along the element tree path; returns `ElementAuthorizationStatus` / `NodePermissions`. Models on `AdvancedContentPermissionService` but uses `UmbracoObjectTypes.Element`/`ElementContainer`, `CalculateElementStartNodeIds`, `RecycleBinElement`.
- `AdvancedElementContainerPermissionService : IElementContainerPermissionService` — resolves container verbs (mapped to canonical) for folders; Authorize Access/Root/Bin only.
- Both registered `AddUnique`; both delegate to the shared element node-permission resolution + storage.

### D4 — UI mirrors the content editor + a generic applicability descriptor
The library editor IS the content editor structurally (one tree-grid, verb columns, split-cell + scope dialog, top-down inheritance). The only deviations are absorbed by a **generic permission-target descriptor** declaring, per `(verb × nodeKind × scope-side)`, whether that cell side is editable or **N/A**. This is the reusable layer for future entities with heterogeneous verb sets. See "Verb model" below.

### D5 — Replace native element + element-folder user-group UI; conditions call our own API
Mirror the document entrypoint pattern, doubled for element + element-folder:
- `exclude()` the 9 `Umb.EntityUserPermission.Element.*` + 5 `Umb.EntityUserPermission.ElementFolder.*` toggle manifests.
- Replace conditions `Umb.Condition.UserPermission.Element` and `Umb.Condition.UserPermission.ElementFolder` with versions that fetch the current user's effective verbs from **our own** management API (not `/user/current`).
- Redirect the native element granular UI (`Umb.UserGranularPermission.Element`) to a "managed in the Library editor" message (reuse `uap-granular-permission-redirect.element.ts`); add a folder redirect only if the folder permission group doesn't already disappear once its toggles are excluded.

### D6 — Element-type create-filtering reuses the Doc Type subsystem
- Add verb `Umb.Element.CreateOfType` + an `ElementTypeVerbs` list. The resolver is verb-aware, so element-type entries coexist with `Umb.Document.CreateOfType` in the existing `DocTypePermission` table without collision.
- Branch `DocTypeCreateContentTypeFilter.FilterAllowedInLibraryAsync` to resolve against the element-type verb.
- Library element-type create-filtering is **section-global** (the platform passes no node context to `FilterAllowedInLibraryAsync`), stored on the virtual root with `ThisNodeAndDescendants`, default Allow (opt-in restriction). The cross-site multi-root argument that keeps doc-type create node-scoped does not apply — the library is a single cross-site section.
- Surface element types (`IsElement && AllowedInLibrary`) in the existing type editor instead of excluding them; render a global toggle for element types vs. the node tree for document types.

### D7 — Defer full type-level verbs for document types
Build everything verb-agnostic, but ship type-level as create-only. Rationale: the one compelling type-level use case (multi-site "may create here") is already covered by node-scoped create-of-type; full type-level Read/Update/Delete introduces a type-vs-node combination semantic (precedence + neutral default) that would touch the working content enforcement path with no concrete use case yet.

### D8 — Editors live in the package's own section
Consistent with the existing Content / Doc Type editors. (Surfacing inside Umbraco's Library section can be a later enhancement.)

## Verb model and applicability matrix

Canonical verbs (stored): the 9 `Umb.Element.*`. Five are shared with folders (Read, Create, Update, Delete, Move); four are element-only (Publish, Unpublish, Duplicate, Rollback). Group them in the UI as the native screens do: **General** (Read, Create, Delete, Publish, Unpublish, Update), **Structure** (Duplicate, Move), **Administration** (Rollback). Folders show only General (Read, Create, Delete, Update) + Structure (Move).

Applicability per `(verb, nodeKind, side)` — what the descriptor encodes:

| Logical verb | Folder: node side | Folder: descendants | Element (leaf): node | Element: descendants |
|---|---|---|---|---|
| Read, Update, Delete, Move | editable (the folder) | editable (sub-folders + elements) | editable (the element) | N/A (leaf) |
| **Create** (parent-oriented) | editable (create children here) | editable (create in sub-folders) | **N/A** (leaves have no children) | N/A |
| Publish, Unpublish, Duplicate, Rollback | **N/A** (folder isn't an element) | editable (elements) | editable (the element) | N/A |

Why this is clean: elements being leaves means **folders + root are the entire inheritance backbone** (exactly the role content nodes play) and **elements only ever hold node-level entries or inherit**. There is no element→element inheritance. `Create` is parent-oriented (checked at the location, never against the created item), so it lives on folders/root and is N/A on leaf elements — same as `Umb.Document.Create` in content.

The frontend `uap-perm-block` already supports split + `na`; the new requirement is that **either side** can be independently N/A, driven by the descriptor.

## Backend work breakdown (test-first)

### Core (`...AdvancedPermissions.Core`)
- Add element verb constants to `AdvancedPermissionsConstants`: `ElementVerbs` (9 canonical), `ElementTypeVerbs` (`Umb.Element.CreateOfType`), `EveryoneDefaultElementVerbs` (e.g. `[Umb.Element.Read]`), and the container→canonical map.
- Reuse `PermissionResolver`/`ResolutionEngine` as-is (no element-specific resolver needed; default state Deny, mirroring content).

### Data (`...AdvancedPermissions.Data`)
- `ElementPermissionEntity` + table `ElementPermission` (columns/indexes mirror `AdvancedPermissionEntity`).
- EF config + `DbSet` in `AdvancedPermissionsDbContext` (+ SqlServer/Sqlite derived classes); update model snapshots.
- `IElementPermissionRepository` + `ElementPermissionRepository` (mirror the content repo; a shared `NodePermissionRepositoryBase<TEntity>` is optional and only worth extracting if content can cheaply adopt it — otherwise accept the small duplication; do not create single-consumer base classes).
- Migrations: `AddElementPermission` for **both** SqlServer and Sqlite, with designer + snapshot updates. Follow ADR-010 invariants.

### Services / enforcement (`...AdvancedPermissions`)
- `ElementPermissionCache` (L1 role entries / L2 user+node resolved; key prefix `UAP.Element.*`).
- Element node-permission service (orchestration: user→groups→context→resolve→cache), reusing `PermissionResolver`.
- `AdvancedElementPermissionService : IElementPermissionService` and `AdvancedElementContainerPermissionService : IElementContainerPermissionService` (the container one maps container verbs → canonical before resolving).
- `ElementPermissionCacheInvalidator` + `ElementPermissionCleanup` — subscribe to element + element-container saved/moved/deleted notifications (exact notification types confirmed in Phase 0; e.g. `ElementDeletedNotification`, `ElementMovedNotification`, container equivalents, `UserGroupDeleted`, `UserSaved`).
- DI: register repository, cache, services (`AddUnique` for the two enforcement services), invalidator, cleanup in `AdvancedPermissionsComposer`.

### Management API
- Element tree controller: root + children across `Element` + `ElementContainer`, annotated with stored entries per role (mirror `AdvancedPermissionsTreeController`; folders have `HasChildren`, elements are leaves).
- Get/save element permission entries for a (node, role).
- **Element effective endpoint** (for the conditions and access viewer) — current user / by-user / by-role effective verbs at a node, covering both element and folder verbs. This is the seam the custom conditions call (the #22351 workaround).
- Element-type create-filtering endpoints: list element types (`IsElement && AllowedInLibrary`), read/save element-type entries (virtual root), audit. Reuse `DocTypePermissionService`/repository/cache.
- Regenerate the OpenAPI client (`Client/src/api/generated`).

## Frontend work breakdown

### Generic shared layer (the reusable investment)
- A **permission-target descriptor**: node kinds, the verb set (with groups), and `appliesTo(verb, nodeKind, side) → editable | na`.
- Extend `uap-perm-block` so each side can be independently `na`.
- Make the scope dialog (`uap-permission-scope-dialog`) applicability-aware: disable/grey a side that is N/A for the current node kind.
- Make `compose-entries`/`decompose-entries` applicability-aware: never emit an entry for an N/A side; reconstruct N/A on read.
- The grid editor + access viewer consume the descriptor (no hardcoded `if (isFolder)`).

### Library editors / viewers
- Library item permissions editor — element tree, the 9-verb descriptor (folders carry inheritance, elements are leaves). Built on the generic components; reuse `tree-ops`, role picker modal, `uap-picker-button`.
- Library access viewer — effective verbs per node with reasoning, reusing `uap-reasoning-dialog`; renders N/A sides with explanatory tooltips.
- Extend the existing type editor to include element types (global toggle) for element-type create-filtering; matching insert-options audit.

### Native UI neutralization (entrypoint)
- `exclude()` the 14 toggle manifests; replace the 2 conditions; redirect `Umb.UserGranularPermission.Element` (reuse the existing redirect element); folder redirect only if needed.

### Manifests, models, localization
- New menu items + workspaces in the package section (library item editor, library access viewer; element-type entries handled in the type editor).
- `element-permission.models.ts` reusing shared `PermissionState`/`PermissionScope`; element verb metadata + descriptor.
- New API wrappers; new localization keys in `en` + `en-us` (other ~20 locales as a follow-up, consistent with PR #22).

## Database / migrations
- New `ElementPermission` table only. No schema change for element-type create-filtering (reuses `DocTypePermission`).
- Seed `$everyone` default `Umb.Element.Read` (+ folder read) at the virtual root, mirroring content's `EveryoneDefaultVerbs`. Verify against the existing seeder / data-import handlers.

## Risks
- **Over-abstraction of the generic frontend layer** — build for the two known shapes (content-uniform; element/folder dual-kind) with clean seams, not a speculative framework.
- **Scope-dialog refactor** is the bulk of the frontend effort and is shared by every cell — treat as foundational.
- **Create simplification** ("one Create column governs both child kinds") should be documented so it isn't surprising.
- **Element UI visibility depends on our own effective API**, not Umbraco's native path — keep that endpoint correct and performant (cache like the document condition does).
