# 03. Backoffice terminology and menu restructure (v18)

**Status**: Implemented on branch (uncommitted) — 2026-06-24
**Date**: 2026-06-21 · **Revised**: 2026-06-24 (reworked post-#35; group headers finalized as "… Permissions" per Option B)
**Target**: Umbraco 18 **only** (v17 explicitly excluded — see "Scope")
**Branch**: `feature/31_rework_grouping_naming` · **Issue**: [#31](https://github.com/Luuk1983/Umbraco.Community.AdvancedPermissions/issues/31)

## Goal

Make the Advanced Permissions backoffice navigation consistent and self-explanatory now that the Library section has doubled the number of screens. The package currently exposes **four editors and four viewers** (eight menu items in the Users section) grouped by *mode* (an "Editors" group and a "Viewers" group), with inconsistent names and four identical `icon-eye` viewers. This restructure regroups everything **by domain**, applies a uniform naming scheme, and gives each item a domain-encoded icon — including the workspace empty-state icons added in #35.

No behaviour, API, persistence, or resolution logic changes. This is navigation, labelling, and iconography only.

## The problem

Under the hood the feature set is a clean **2 × 2 × 2 matrix**:

| | **Node permissions** (Allow/Deny verbs on tree nodes) | **Type-level permissions** (currently create/insert filtering; future of-type verbs) |
|---|---|---|
| **Content domain** | Content Permissions Editor → Access Viewer | Document Type Permissions Editor → Insert Options Viewer |
| **Library domain** | Library Permissions Editor → Library Access Viewer | Library Element Type Permissions → Library Insert Viewer |

= 2 domains × 2 aspects × 2 modes = 8 leaf screens. The menu groups them by *mode*, forcing the user to mentally re-pair each viewer with its editor across two groups.

Concrete terminology/iconography defects today:

1. Inconsistent "Editor" suffix — three editors have it; "Library Element Type Permissions" does not.
2. Inconsistent domain prefix — "Content Permissions Editor" vs. its viewer "Access Viewer" (no domain), while Library prefixes both sides.
3. Editor/viewer vocabulary mismatch — "Document Type Permissions" pairs with "Insert Options Viewer"; "Library Element Type Permissions" pairs with "Library Insert Viewer".
4. "Insert" (content) vs. "Create" (library) for the same concept.
5. All four viewers share `icon-eye` — visually indistinguishable (both in the menu and in their new empty-state panels).
6. "Permissions" is overloaded — node verbs in some names, type-creation in others.

## Design decisions

- **D1 — Group by domain, not by mode.** Replace the "Editors"/"Viewers" groups with four domain groups. Rationale: pairs the viewer with its editor and matches how users think.
- **D2 — Editor and viewer stay as separate menu items (not workspace tabs).** Rationale: users frequently want the editor and its viewer open side by side; separate items allow that, workspace tabs would not.
- **D3 — Four symmetric groups, short item labels.** Group headers carry the domain; item labels are just the function. Accepted trade-off: the label "Permissions Editor" repeats across groups (the header + icon disambiguate).
- **D4 — Type-level aspect is NOT named "insert"/"create".** These editors today filter the create lists via a package-invented verb, but are intended to gain real of-type verbs (`DeleteOfType`, `MoveOfType`). Names must not bake in "insert"/"create". Result: every editor is **"Permissions Editor"**, every viewer is **"Access Viewer"** — both survive the future verbs.
- **D5 — Domain-encoded icons (Scheme A).** Each item's icon is its domain glyph; editor and viewer in a group share it. The four viewers stop being identical eyes. All four glyphs verified present in the Umbraco icon registry (`icon-document`, `icon-diploma`, `icon-globe`, `icon-thumbnail-list`).
- **D6 — Full descriptive page headlines.** Menu labels are short ("Permissions Editor"), but each workspace's H1 keeps the **full** name ("Content Permissions Editor", "Document Type Access Viewer", …).
- **D7 — Internal aliases / entityTypes unchanged.** Identifiers like `uap-doc-type-create-audit` and `UAP.Workspace.AccessViewer` are never user-visible; renaming is pure churn. Only user-facing labels, icons, and grouping change.
- **D8 — Workspace empty-state icons stay in sync with menu icons (NEW, post-#35).** #35 reworked every workspace to render a shared empty-state "selection panel" (`help/uap-selection-panel.element.ts`) with a `ctaIcon`. Each screen therefore has **two** icon sites — `menuItem.icon` and `ctaIcon` — which must always show the same domain glyph. The Scheme A change updates both.
- **D9 — Group headers carry purpose, not just domain (Option B).** Bare domain headers (Content, Document Types, …) inside the native *Users* section don't convey that the groups are about permissions — the top-nav section name says "Users". So each header is **"&lt;Domain&gt; Permissions"**: "Content Permissions", "Document Type Permissions", "Library Permissions", "Library Element Type Permissions". Items remain "Permissions Editor" / "Access Viewer". (A dedicated top-level "Advanced Permissions" section was considered — cleanest context — but rejected to avoid the per-user-group section-access step and to keep the tools under Users. A future "single workspace, editor+viewer as two views" model was also noted and deferred.)
- **D10 — Native localization keys: only for single native terms, not composites.** A manifest `label` references exactly one localization key, and dictionaries don't interpolate key-in-key, so composite headers ("Content Permissions") must be our own per-locale strings (we briefly switched the 3 bare headers to `#sections_content` etc., then reverted under D9). Reusing native keys for pure shared terms, and the "native-noun + per-locale template" composition technique for component-rendered composites (headlines/intros/redirects), is deferred to a separate **follow-up issue** (terminology alignment).

## Final backoffice layout (Users section)

```
USERS                        ← native, unchanged
  Users
  User Groups

CONTENT PERMISSIONS              ← UAP.SidebarApp.Content            (weight 60)
  Permissions Editor   icon-document
  Access Viewer        icon-document

DOCUMENT TYPE PERMISSIONS        ← UAP.SidebarApp.DocumentTypes       (weight 55)
  Permissions Editor   icon-diploma
  Access Viewer        icon-diploma

LIBRARY PERMISSIONS              ← UAP.SidebarApp.Library             (weight 50)
  Permissions Editor   icon-globe
  Access Viewer        icon-globe

LIBRARY ELEMENT TYPE PERMISSIONS ← UAP.SidebarApp.LibraryElementTypes (weight 45)
  Permissions Editor   icon-thumbnail-list
  Access Viewer        icon-thumbnail-list
```

Each group = one `sectionSidebarApp` (kind `menu`) + one `menu` holding its two `menuItem`s — the same mechanism the current "Editors"/"Viewers" groups use, scaled from two groups to four. The eight workspaces and the Help modal (`UAP.Modal.Help`) are otherwise untouched; only the menu they hang under, their label, and their two icon sites change. The outdated "grouped by workflow (editing vs inspecting)" comment at the top of the sidebar-app block in `manifests.ts` is updated to describe domain grouping.
